# 🚨 考场实操与大题 0分 踩坑实录

## 🛑 环境致命坑：千万别只双击 .c 文件！
* **血泪教训**：在机考环境（如 Visual Studio 或 VC++ 2010）中，直接打开 `.c` 源文件是无法编译运行的（运行按钮是灰色的，或者直接报错）。
* **唯一正确姿势**：必须在文件夹里找到并双击打开 .sln (Solution 解决方案) 文件（老版本可能是 .dsw 工作区文件）。只有打开整个工程，IDE 才能正确链接上下文，才能正常按 F7 编译和 Ctrl+F5 运行验证结果。

## ❌ 核心代码的 5 个“语法死穴”（以求平均值题为例）

### 1. 变量“黑户” (Undeclared Identifier)
* **致命原因**：C 语言极其死板，所有变量出生前必须“上户口”。绝不能习惯性地直接写 n = 0; sum = 0;。
* **正解**：int i, n = 0; double sum = 0.0; （连 for 循环里的 i 也必须在最前面提前声明！）

### 2. 整数除法塌陷
* **致命原因**：在 C 语言中，整数除以整数（如 5 / 2），结果直接是 2，后面的小数部分会被瞬间丢弃。
* **正解**：保存总和的变量一开始就必须声明为浮点型 double sum = 0.0;。

### 3. 大括号作用域错乱
* **致命原因**：没有大括号 {}，if 只能管住紧挨着它的第一行代码。如果你写成：
    if (x[i] % 2 == 0)
        n++;
    sum += x[i];
  结果就是：n 确实只数了偶数的个数，但 sum 却无脑把所有的奇数和偶数全加了一遍，逻辑彻底跑偏。
* **正解**：必须用大括号把相关操作绑死在 if 内部！
    if (x[i] % 2 == 0) {
        n++;
        sum += x[i];
    }

### 4. 指针赋值没加 *
* **致命原因**：参数传进来的是 double *ave。ave 本身是个地址（相当于储物柜的门牌号）。你不加 * 直接赋值（ave = sum / n;），等于强行把算出来的平均分变成了一个新的门牌号，主函数根本拿不到真实的数据。
* **正解**：*ave = sum / n;（加上 *，代表打开这个门牌号对应的储物柜，把成绩放进去）。

### 5. 答应了返回值却放鸽子
* **致命原因**：函数头明明写着 int fun(...)，代表承诺返回一个整数，但在函数最后一行却什么都没写。主函数写着 n = fun(...) 眼巴巴等着接收结果。你不给 return，编译器直接判定你语法错误，0分处理。
* **正解**：结尾必须加上 return n;。
